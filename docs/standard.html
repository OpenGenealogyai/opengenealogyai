1. What Is MAXGEN?

MAXGEN is an open JSON standard for representing individual genealogical identities in a way that is honest about uncertainty.

Some or most genealogy platforms today force researchers to pick one answer. When two records disagree about who someone's father was, one record wins and the other disappears. MAXGEN rejects that model.

In MAXGEN, every claim is an assertion — it has a confidence score, a source, and a timestamp. Nothing is ever overwritten. Multiple possible parents can coexist. Conflicting birth years both survive. The uncertainty is the data.

An assertion with low confidence and a real source is worth more than a certain-looking answer with no source.

3. The Three Record Types

MaxGen Person

The Individual

A probabilistic human identity synthesized from one or more source documents. Every field is an assertion array — never a single value. Multiple possible parents can coexist with individual confidence scores.

person.schema.json · .maxgen
MaxGen Record

The Source Document

A single original historical document exactly as found — birth certificate, census row, parish register, obituary, land patent. No interpretation is added at this layer. Every MaxGen Person assertion traces back to at least one MaxGen Record.

raw-record.schema.json · .mxrecord
MaxGen Task

The Work Queue Entry

A unit of distributed work — scraping, OCR extraction, embedding, or verification. Used by the contributor checkout system and AI agents. Enables fully distributed, autonomous database construction.

task-queue.schema.json

MaxGen Person — Key Design Decisions

  • composite_confidence — overall probability that all assertions in this record refer to the same real individual
  • parent_assertions — multiple entries are normal; confidence expresses certainty, not conflict; neither assertion is deleted
  • conflict_flag — marks when two assertions for the same role disagree; both survive
  • is_living — when true, the entire record returns 404 on all public endpoints
  • redistribution_license — most restrictive license among all contributing source records

MaxGen Record — License Tiers

  • CC0 / public-domain — fully open, included in all datasets
  • CC-BY / CC-BY-SA — open with attribution or share-alike requirements
  • tier2-private — never included in open datasets; returned only to authenticated owner

4. The Confidence Score

Every assertion in MAXGEN carries a confidence value from 0.0 to 1.0. Scores are mandatory and must be honest. A consistent 0.4 is more valuable than an inflated 0.9.

ScoreVisualMeaning
0.9 – 1.0
Field appears verbatim in source document, no ambiguity
0.7 – 0.89
Minor OCR error or spelling variation from the original
0.5 – 0.69
Inferred from context — not stated explicitly in the source
0.3 – 0.49
Ambiguous or physically damaged source document
0.0 – 0.29
Best guess only — minimal documentary support

Agents and contributors are expected to score conservatively. Over-confident scores degrade the quality of every downstream research report that relies on this assertion.

5b. Contributor Attribution — Required Field

MaxGen has two levels of contributor attribution that work together:

  • MaxGen Record — carries a single contributor object: the person who extracted and embedded that specific source document.
  • MaxGen Person — carries a contributors array: every person who contributed any source record linked to this individual, aggregated automatically. If 15 different contributors each embedded one document about Richard Maxwell, Richard Maxwell's record lists all 15.

This means contributor attribution is always visible wherever genealogy data is displayed — not just in a site-wide footer. When a user views any ancestor, a persistent dropdown lists every contributor to that person's data by name. Attribution is permanent and cannot be removed after a record is published.

Per-Record: contributor object (MaxGen Record)

FieldTypeRequiredDescription
display_namestringYesPublic name shown in attribution credits
contributor_idstringYesUnique stable ID for payout tracking
contribution_dateISO 8601 dateYesDate the record was submitted
verified_bystringNocontributor_id of the quality checker
verification_dateISO 8601 dateNoDate verification was completed

Per-Person: contributors array (MaxGen Person)

Each entry in the array is a contributor object extended with two additional fields:

FieldTypeDescription
(all fields above)—Inherited from the source record's contributor object
source_record_idstringThe MaxGen Record this contributor is credited for
assertions_contributedstring[]Which fields on this person this record helped establish (e.g. ["birth_year","birth_place"])

Display rule: Any interface that renders MaxGen data — a web viewer, a research report, a mobile app — must provide a persistently accessible contributor dropdown for every person displayed. The first row shows the most recent or most prolific contributor. All others are one tap away. This is not optional.

Records without a valid contributor object are rejected by the Layer 0 schema validator and will not be accepted into the database.

6. File Extensions

MAXGEN uses two file extensions to distinguish record types at a glance:

  • .maxgen — MaxGen Person records: william_whitfield_1842.maxgen
  • .mxrecord — MaxGen Record source documents: 1880_census_ohio_p42.mxrecord

MaxGen Task records live in the pipeline work queue and are not typically stored as standalone files.

7. Versioning

This is MAXGEN v1.0. The schema_version field in every record pins the version at write time, ensuring records remain readable as the standard evolves.

  • Breaking changes — increment the major version (1.0 → 2.0)
  • Additive changes — increment the minor version (1.0 → 1.1)

Records written to v1.0 will always be interpretable as v1.0 records. No silent migration.

8. Why JSON?

  • Human-readable and machine-readable — no proprietary parser required
  • Natively supported by every modern programming language
  • Vector-embeddable — a MAXGEN record can be fed directly to an embedding model without conversion
  • GitHub-friendly — diff, review, and merge genealogy records exactly like source code
  • Schema validation — JSON Schema 2020-12 enforces structure at ingest time
  • No lock-in — export your data at any time with zero data loss

10. Governance — How the Standard Changes

MaxGen is CC0 public domain — anyone can use it. But the canonical version of the standard is stewarded by Garlon Maxwell, who holds final approval on all schema changes. This protects the standard from fragmentation while keeping it open to community input.

How to Propose a Change

All proposals begin as a GitHub Issue in the opengenealogyai repository. Use the Schema Change Proposal issue template, which asks for:

  • Problem — What real use case does the current schema fail to support?
  • Proposed change — The new or modified field(s), with type and description.
  • Backward compatibility — Does this break existing records? How should they be migrated?
  • Examples — At least one before/after JSON snippet showing the change in practice.

The Review Process

  1. Community discussion — Anyone can comment on the issue. Discussion is open for a minimum of 7 days for minor changes, 30 days for breaking changes.
  2. Steward review — Garlon Maxwell reviews the proposal and discussion. He may request changes, ask for more examples, or put the proposal to a broader vote.
  3. Approval or rejection — Garlon posts a final decision with reasoning. Nothing merges without his explicit sign-off.
  4. Implementation — Approved changes are merged into the schema files by the Data Quality Agent or a designated contributor.

Versioning Rules

Change TypeVersion BumpExample
New optional fieldv1.0 → v1.1Adding burial_county to a RawRecord
New required fieldv1.0 → v2.0Making source_url required
Renamed or removed fieldv1.0 → v2.0Renaming conf to confidence
Bug fix with no field changev1.0 → v1.0.1Fixing a regex in the validator

Records written to any version of MaxGen remain readable forever. There is no forced migration. A v1.0 record processed by a v2.0 tool must degrade gracefully — never silently fail.

Governance Principle

"MaxGen is open because anyone can use it and build on it. It is not open to fragmentation. One canonical version. One steward. Every change earns its place in the record."

— Garlon Maxwell, Steward

What Garlon Does Not Control

Garlon controls the canonical schema — the official version. He does not control how you use it. You can:

  • Fork the schema and add your own fields for private use — no permission needed.
  • Build tools that extend MaxGen with additional metadata.
  • Publish your own MaxGen-compatible records without approval.
  • Propose changes to the canonical standard via GitHub Issues — all proposals are welcome.

What you cannot do is call your fork "MaxGen" if it diverges from the canonical version. The name means something specific.

Discussion Forum

For broader discussion about genealogy standards, schema design, and open data — outside of specific change proposals — use the GitHub Discussions board. This is where the community debates ideas before they become formal proposals.
